## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4dp | Icon-to-label gaps, dense chip internal padding |
| sm | 8dp | Compact element spacing, inner card padding |
| md | 16dp | Default horizontal screen margin (Material3 standard), list item padding |
| lg | 24dp | Section padding, top/bottom screen padding |
| xl | 32dp | Layout gaps between major sections |
| 2xl | 48dp | Major section breaks |
| 3xl | 64dp | Page-level spacing (e.g., bottom padding above nav bar) |

Exceptions:
- Bottom navigation bar height: 80dp (Material3 `NavigationBar` default — do not override)
- Touch targets: 48dp minimum on all tappable icons (Material3 accessibility minimum)
- FAB size: 56dp (Material3 standard FloatingActionButton)

Source: Material3 layout guidelines; md=16dp is the canonical horizontal screen margin for Android.

## Typography

Material3 type scale roles used in Phase 1 (screen stubs use only these roles — no custom text):

| Role | Size | Weight | Line Height | M3 Token | Usage |
|------|------|--------|-------------|----------|-------|
| Display | 28sp | 400 (Regular) | 1.2 | `titleLarge` | App bar / screen titles |
| Heading | 20sp | 500 (Medium) | 1.3 | `titleMedium` | Section headings, bottom nav labels when selected |
| Body | 16sp | 400 (Regular) | 1.5 | `bodyLarge` | Primary body content, list item primary text |
| Label | 12sp | 500 (Medium) | 1.4 | `labelMedium` | Bottom navigation labels, chip text, metadata |

Rules:
- Maximum 4 type roles in use across the entire app; no ad-hoc sizes outside this scale.
- Weight is exactly 2 values: 400 (Regular) and 500 (Medium). No bold (700) in Phase 1 stubs.
- sp units only (scales with system font size — accessibility requirement on Android).

Source: Material3 type scale (m3.material.io/styles/typography/type-scale-tokens). Phase 1 stubs only show screen titles and nav labels, so only Display and Label are actively needed; Body and Heading declared now for consistency across phases.

## Color

Material3 dynamic color is NOT used in Phase 1 (requires Android 12+, min SDK is 26 per RESEARCH.md, and dynamic color requires explicit opt-in). A static seed color scheme is defined here and applied via `MaterialTheme(colorScheme = ...)`.

Seed color: **#1A73E8** (Google Blue — familiar, high contrast, professional; suitable for a business card app).

| Role | Light Mode Value | Dark Mode Value | Usage |
|------|-----------------|-----------------|-------|
| Dominant (60%) — `surface` | #FFFBFE | #1C1B1F | Screen backgrounds, scaffold backgrounds |
| Secondary (30%) — `surfaceVariant` / `secondaryContainer` | #E7E0EC | #3E3D42 | Bottom nav background, card surfaces, list item backgrounds |
| Accent (10%) — `primary` | #1A73E8 | #8AB4F8 | FAB, active nav indicator, selected state highlight |
| Destructive — `error` | #B3261E | #F2B8B5 | Delete confirmations only |

Accent reserved for:
1. Floating Action Button (FAB) fill color on CardListScreen
2. Active bottom navigation item indicator (selected state background pill)
3. Active bottom navigation item icon tint (selected state)

No other elements use `primary` / accent color in Phase 1 stubs.

Dark mode: `darkColorScheme {}` is declared in `Theme.kt` from Phase 1 using the same seed. The `isSystemInDarkTheme()` check selects the correct scheme at runtime. Dark mode must be functional from day one — not deferred.

Source: Material3 color roles (m3.material.io/styles/color/roles). Seed color is author's discretion (no color specified in CONTEXT.md or REQUIREMENTS.md).

## Navigation Shell Contract

Phase 1 establishes the navigation shell that all subsequent phases build on. Decisions made here cannot be changed without reworking all screens.

### Pattern: Bottom Navigation Bar

Use Material3 `NavigationBar` (bottom bar) — not a Navigation Drawer. Rationale: single-hand reachability on mobile; standard pattern for 3–5 top-level destinations; consistent with Android HIG.

Bottom navigation contains exactly **3 items** (top-level destinations only):

| Position | Label | Icon | Route | Active Phase |
|----------|-------|------|-------|--------------|
| 1 | Cards | `Icons.Rounded.Style` | `CardListRoute` | Phase 1 stub → Phase 3 real |
| 2 | Scan | `Icons.Rounded.CameraAlt` | `ScanRoute` | Phase 1 stub → Phase 2 real |
| 3 | Tags | `Icons.Rounded.Label` | `TagManagerRoute` | Phase 1 stub → Phase 3 real |

`CardDetailRoute` and `OcrReviewRoute` are **not** bottom nav items — they are sub-screens reached by in-stack navigation from their parent screens.

### FAB on CardListScreen

A `FloatingActionButton` with `Icons.Rounded.CameraAlt` appears on `CardListScreen`. Tapping it navigates to `ScanRoute`. In Phase 1 this is a stub — the FAB exists, tap does nothing beyond nav. Label: none (icon-only FAB, 56dp).

### App Bar

`TopAppBar` with `titleLarge` text on all five screens. Back navigation arrow appears on `CardDetailScreen`, `OcrReviewScreen`, and `TagManagerScreen` (screens not reachable from bottom nav directly). `CardListScreen` and `ScanRoute` do not show a back arrow (they are top-level nav bar destinations).

### Screen Stub Layout

Each stub screen renders only:
1. A `Scaffold` with the app-level `TopAppBar` and `NavigationBar`
2. A centered `Text` composable with the screen name using `titleLarge` style
3. No other content

```
┌─────────────────────────────┐
│  TopAppBar [Screen Name]    │  ← titleLarge, surface background
├─────────────────────────────┤
│                             │
│                             │
│       "Screen Name"         │  ← centered, titleLarge
│                             │
│                             │
├─────────────────────────────┤
│  Cards │   Scan   │  Tags   │  ← NavigationBar, 80dp
└─────────────────────────────┘
```

## Theme File Structure

Phase 1 creates these files. All subsequent phases import from here — never redeclare tokens inline.

| File | Contents |
|------|----------|
| `ui/theme/Color.kt` | `val Blue40`, `val Blue80`, `val Error40`, `val Error80`, and all M3 color role assignments |
| `ui/theme/Type.kt` | `val AppTypography = Typography(...)` declaring the 4-role type scale |
| `ui/theme/Theme.kt` | `CardKeeperTheme(darkTheme, content)` composable using `lightColorScheme` / `darkColorScheme` |

`MaterialTheme` is wrapped in a single `CardKeeperTheme {}` composable. Every screen is wrapped in `CardKeeperTheme`. No screen applies `MaterialTheme` directly.
